03

The change

src/core/agents/pi.ts

● before — discards valid output

// reads ONLY the last assistant message
const finalText = textFromAssistantMessage(
  latestAssistantMessage).trim() || …;

try {
  parsed = JSON.parse(finalText); // 💥 prose
} catch (err) {
  reject(new Error(
    `Failed to parse pi output: …`));
}

● after — recover over all turns

const parsed = recover(finalizedSegments);
if (parsed === null) {
  reject(new Error(
    `Failed to parse pi output: …`));
  // genuine failures still surface
  return;
}
const output = validateAgentOutput(parsed, schema);
resolve({ output, usage: lastEmittedUsage });

● the two-pass recovery (pure function — unit-testable)

const accepts = (v) => { try { validateAgentOutput(v, schema); return true; } catch { return false; } };

function recover(segments) {
  // Pass A — a whole turn that is pure JSON (after fence-strip)
  for (let i = segments.length - 1; i >= 0; i--) {
    try { const p = JSON.parse(stripJsonFences(segments[i])); if (accepts(p)) return p; } catch {}
  }
  // Pass B — an object embedded in prose, rightmost-in-turn
  for (let i = segments.length - 1; i >= 0; i--) {
    const found = extractLastJsonObject(segments[i], accepts);
    if (found) return JSON.parse(found);
  }
  return null;
}
A

Pure-JSON turn

Scan turns last→first; a turn whose entire text is the JSON object is the strongest terminal signal. Catches the real bug.

B

Embedded object

Only if no pure-JSON turn exists: dig the rightmost balanced object out of prose. So a fenced "example" in a summary can never beat a clean final turn.

● the accumulator — resolves Codex's blocking objection

  • Accumulate text per message, keyed by pi's stable message id (messageKey()) — fall back to a monotonic synthetic key allocated on assistant message_start/turn_start (lazily on first text_delta). Never key off contentIndex or role alone.
  • text_delta → append to that message's block; text_end → set the block.
  • On assistant message_end/turn_end → join blocks in contentIndex order, push once to finalizedSegments (dedupe via a finalizedKeys Set).
  • Close-time guard (Codex's add): finalize any still-active accumulator even if no text_end/message_end ever arrived. Don't depend on latestAssistantMessage text existing.
  • Keep the existing agent_end messages-array fallback. Never include tool output.

Untouched: stopReason === "error"/"aborted" handling · all TokenUsage accumulation · buildPiPrompt.

04

Tests — TDD, red first

recover(segments) — pure fn

  • prose + JSON + prose
  • pure-JSON turn
  • JSON then later prose turns (the real bug)
  • trailing fenced "example" → Pass A still picks the real earlier turn
  • no JSON at all → null
  • multiple valid turns → most-recent wins

accumulator — synthetic events

  • contentIndex 0 reused across turns → no collision
  • duplicate text_end + message_end → finalized once
  • text_delta with no end event → finalized at close
  • JSON in block 1 + prose in block 2 → ordered join

e2e only if a pi mock is cheap (none today) — not blocking.

05

Out of scope & residual risk

Not touching the prompt, and not adding an "after the last tool call" heuristic — Codex noted that would break the real case, since the valid JSON legitimately precedes a later tool call.
Residual risk (low, accepted): if a trailing prose turn were entirely a valid schema object on its own, Pass A could pick it over an earlier real one. The two-pass design already eliminates the realistic case (a fenced example embedded in prose).
